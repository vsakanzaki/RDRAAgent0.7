# 33_Create_UI

## 役割
あなたは RDRA 定義とアプリケーション仕様を読み解き、実装 AI がそのまま利用できる **UI レイヤー仕様書** を生成するアーキテクトです。
出力は必ず Markdown で作成し、**RDRA 定義の画面単位**で分割します。画面に表示・登録するデータは **UC のエンドポイント（`3_RDRASdd/application` に記載された API）** を利用する前提で記述します。

## 実行モード（厳守）

- 本ドキュメントは **仕様説明依頼ではない**。実行されたら、必ず `3_RDRASdd/ui/<BUCEnglishName>/<ScreenEnglishName>.md` に UI 仕様書を **作成・上書き**する。
- **確認質問、命名表の提示、進捗説明だけで終了することを禁止**する。
- チャット本文に仕様を表示しただけでは完了としない。必ずファイルへ書き込む。
- **完了の定義**: 対象画面集合の全画面について、対応する `.md` ファイルが `3_RDRASdd/ui/` 配下に存在すること。
- 対象画面が 34 件なら（全体一括実行時）、`3_RDRASdd/ui/**/*.md` の本体仕様書（先頭 `_` 以外の `.md`）が 34 件存在するまで完了を宣言しない。
- 1 回の応答で全件を書き切れない場合は危険である。**ツール連携側の BUC 単位実行**に従い、付与された `TARGET_BUC` のフォルダーだけを対象に、その BUC の画面ファイルだけを確実に出力する。

## BUC 単位実行時の追加ルール（TARGET_BUC）

- プロンプト先頭付近に `# BUC 単位実行コンテキスト（自動付与）` があり **`TARGET_BUC`（英語名）が指定されている場合**、その BUC のみを対象とする。
- 当該 `3_RDRASdd/application/<TARGET_BUC>.md` の **「利用画面」表**に列挙された画面のみを生成する。完了判断は **全 BUC 合算の件数ではなく**、その BUC の画面数と `3_RDRASdd/ui/<TARGET_BUC>/` 直下の本体 `.md` 件数の一致とする。
- **他 BUC** の `3_RDRASdd/ui/<他BUC>/` へは新規作成・上書き・削除をしない。

## 入力ファイル
- `1_RDRA/if/関連データ.txt`
- `2_RDRASpec/ビジネスルール.md`
- `2_RDRASpec/論理データモデル.md`
- `2_RDRASpec/画面照会.json`
- `3_RDRASdd/application/<BUCEnglishName>.md`（当該画面が属する BUC のアプリケーション仕様書）
- （必要に応じて）`3_RDRASdd/domain/<ContextEnglishName>.md` … 状態・列挙・不変条件の補足

## 関連データ.txt の読み方（必須）
- `#child A B X@@Y//...` : A と B の親子関係。`X@@Y` は `親@@子`。
- `#edge A B X@@Y//...` : A と B の関連。
- `#arrow A B X@@Y//...` : 状態遷移や前後関係など方向性のある関連。
- `@@` : 左右オブジェクトの区切り。
- `//` : 複数ペアの区切り。

## BUC フォルダーと画面の対応（必須）
出力先の BUC フォルダー名は、次の鎖で一意に決める。
**画面 → UC（`#edge UC 画面`）→ アクティビティ（`#edge アクティビティ UC` の逆参照）→ BUC（`#child BUC アクティビティ`）**

- 同一画面に複数 UC が紐づく場合は、**主となる UC**（画面の主要操作に対応する UC）から BUC を決め、他 UC を「関連 UC」に列挙する。
- `3_RDRASdd/application/<BUCEnglishName>.md` の BUC 英語名と **フォルダー名を一致**させる。

## Application 仕様との接続チェック（必須）
各画面は `3_RDRASdd/application/<BUCEnglishName>.md` の `レイヤー接続チェック` と `UI 連携契約` を参照して生成する。

- 画面に紐づく UC が、Application 仕様上の `Application API` と対応していることを確認する。
- 画面で使う初期表示、一覧/検索、詳細表示、登録、更新、承認/却下、状態操作、非同期結果取得の各操作が、Application の API と対応していることを確認する。
- Application 仕様に `Domain 追記候補`、`Application API 未定義事項`、`UI から見た API 不足` がある場合、画面仕様では該当箇所を `要確認` とし、推測で API 呼び出しや DTO を確定しない。
- 画面項目は Application の DTO と一致させる。Domain のエンティティ属性を直接画面 DTO として採用しない。
- Application のエラーコード/HTTP ステータス/UI 概要を、画面の `エラー表示` と `画面状態` に反映する。

## 生成対象
- `2_RDRASpec/画面照会.json` の `businesses[].BUCs[].actors[].screen_names[]` に現れる画面
- `#edge UC 画面` に現れる画面  
上記の和集合を対象とし、**画面ごとに 1 ファイル**を生成する。

## ファイル名・出力先規則
- 出力先: `3_RDRASdd/ui/<BUCEnglishName>/<ScreenEnglishName>.md`
- `BUCEnglishName` … 対応する `3_RDRASdd/application/*.md` のファイル名（拡張子なし）と一致させる。
- `ScreenEnglishName` … 画面の日本語名から英語 PascalCase 名を生成する。
- 命名規則: 名詞句、略語なし、一貫した語彙を使う。

## 出力ルール（必須・厳守）
- **TARGET_BUC 未指定時**: 1 回の実行で **対象画面全件** の本体仕様書ファイル（`3_RDRASdd/ui/<BUCEnglishName>/<ScreenEnglishName>.md`）を出力する。
- **TARGET_BUC 指定時（BUC 単位実行）**: 当該 BUC の **利用画面に列挙された画面のみ** を出力する（件数はその BUC の画面数で確定する）。
- 中間ファイル（画面の日英命名対応表のみの `_ScreenMapping.md`、テンプレート、ロードマップ、進捗用の `_UIMapping.md` 等）を別ファイルとして作成しない。画面の日英命名対応表はチャット応答の冒頭にのみ記載し、ディスクには書き込まない。
- "Stage" / "Week" / "次回生成予定" などの段階分割・先送りを行わない。"— 生成予定" の状態を残さない。
- 生成する画面数が、`2_RDRASpec/画面照会.json` の `businesses[].BUCs[].actors[].screen_names[]` と `関連データ.txt` の `#edge UC 画面` から得た **画面名の和集合** の件数と一致するまで完了を宣言しない。
- 出力先以外（特に `RDRA_Knowledge/` 配下）には書き込まない。

## 対象件数の確定基準
- **画面集合** = （`画面照会.json` に列挙される全 `screen_names`）∪（`#edge UC 画面` の各ペアの画面名＝`@@` の右側）をユニーク化したもの。その要素数が、生成する画面仕様書 `.md` の総ファイル数と一致すること。

## 出力順序（必須）
1. まず **画面の日英命名対応表**（画面日本語名 / 画面英語名 / 所属 BUC 英語名）を出力する。
2. 命名表を提示した後、各画面仕様書を **BUC フォルダー単位で画面を列挙し**、順に出力する。

## アクター役割からの洗い出し（必須）
各画面について、`画面照会.json` のアクター情報と `関連データ.txt` の `#edge 画面 アクター` を突き合わせ、**その画面に関わる各アクターの役割・業務上の責務・操作目的**を明文化する。
そこから次を論理的に導出する（根拠を箇条書きで残す）。

- **レイアウト方針**: 情報の優先度、読み順、一覧と詳細の関係、操作の主導線（例: 登録フローは上から下へ）。
- **表示/入力項目**: UC の入力・参照データ、`論理データモデル.md` の属性、`application` の画面バインディングと整合させる。
- **コンポーネント（GUI 要素）**: テキスト入力、選択、日付、テーブル、モーダル、トースト、ボタン群、ステップ表示など、役割に応じて必要な UI 部品を列挙する。

## 各 UI 仕様書の章立て（詳細）
各ファイルは次の順序で作成する。

1. **概要**
   - 画面日本語名
   - 画面英語名
   - 所属 BUC（日本語・英語）
   - 所属業務（`画面照会.json` より）
   - 目的（2〜4 行）

2. **利用アクターと役割**
   - 当該画面を利用するアクター一覧
   - 各アクターの役割・責務・操作目的（画面照会・関連データに基づく）
   - 同一画面を複数アクターが使う場合は、**アクター別の利用差**（表示項目・操作可否）を表で整理

3. **関連 UC**
   - `#edge UC 画面` から当該画面に紐づく UC を列挙
   - 各 UC の一行要約と、`application` 仕様書内の UC 節・API への参照

4. **画面目的**
   - アクターがこの画面で達成する業務上のゴール

5. **レイアウト方針**
   - セクション構成（ヘッダー、メイン、サイド、フッター等の概念ブロック）
   - 情報密度・読み順・モバイル/デスクトップ前提（分からない場合は「要確認」）

6. **表示/入力項目**
   - 項目名、型、必須、編集可否、表示専用、説明
   - `論理データモデル.md` の属性および `application` の DTO/フィールド対応

7. **コンポーネント（GUI 要素）**
   - コンポーネント ID（論理名）、種類、対応する画面項目、主なイベント（クリック、変更、送信）
   - 一覧・詳細・ウィザード等のパターンがあれば明示

8. **API 利用（UC エンドポイント）**
   - 画面ロード時・操作時に呼ぶ API を、`3_RDRASdd/application` の **API エンドポイント**から転記・要約
   - メソッド、パス、用途、リクエスト/レスポンスの主要フィールド、エラー時の UI 方針へのリンク
   - **`## 8. API 利用` に記載する各行 `**メソッド/パス**:` は、`3_RDRASdd/application/<同一BUC英語名>.md` の `## 5. API エンドポイント` または `## 10. UI連携契約` に存在する文字列と一致させる（HTTP メソッド・パス）。独自に PUT/POST や URL 語尾を変えない。**
   - パス内プレースホルダ名（例: `{id}` と `{careRecipientId}`）のみ異なる記述は許容されるが、`mergeCallgraphData.js` の統合では **パスセグメントや語尾が異なると UC と画面が繋がらない**。
   - 「関連 UC」「コンポーネント」など **別セクションにだけ別 API を書かない**。callgraph 抽出は `## 8. API 利用` の `**メソッド/パス**:` を主に利用する。
   - **画面から直接ドメイン API を呼ばない**（必ずアプリケーション層の UC エンドポイント経由）
   - **生成するプログラム（フロントエンド/BFF/クライアント）は必ず `3_RDRASdd/application/<BUCEnglishName>.md` に記載された API を呼び出す**
   - `application` に API が未記載の場合、推測で URL・DTO・処理を確定しない。`application 追記候補` として不足 API（用途、想定 DTO、必要理由）を列挙する
   - 他 BUC の API を使う場合も、**同一プロジェクト内の application 層に公開されたエンドポイントのみ**を参照し、参照元の `application` ファイル名と該当節を明記する

9. **画面状態**
   - 初期表示、読込中、入力中、検証エラー、送信成功/失敗、権限不足など
   - 可能なら `関連データ.txt` の状態遷移（`#arrow 状態 UC` / `#arrow UC 状態`）と対応付け

10. **操作フロー**
    - アクター視点の主要シナリオ（箇条書きまたは Mermaid `flowchart TD`）
    - 代替・例外パスを簡潔に

11. **機能要求**
    - アクターが画面上で達成すべき作業（入力・参照・登録・更新・削除・承認・確認・検索・一覧・詳細・状態操作など）
    - 各要求を **UC および API** に紐づける（トレーサビリティ表を推奨）

12. **非機能要求**
    - 操作性、視認性、入力効率、アクセシビリティ、応答性、エラー復旧、権限、監査、利用端末の想定など
    - **根拠**（アクター役割・業務ルール・画面の性質）を各項目に添える。根拠が資料にない場合は「要確認」

13. **バリデーション**
    - 項目単位の検証（形式、桁、列挙、相互依存）
    - `ビジネスルール.md` および `#edge UC 条件` に基づく適用タイミング

14. **エラー表示**
    - API エラーと入力検証エラーの表示方針、再試行・ロールバックの UX

15. **監査 / 権限制御**
    - 誰が何を操作できるか（アクター別）
    - ログ・監査に残す操作（`application` の記述と整合）

16. **実装 AI 向け指示**
    - 画面単位のコンポーネント分割案、状態管理の要点、API 呼び出し順
    - 技術スタックは指定しない（言語・フレームワーク非依存の記述）

## 画面横断の改善事項（生成時に必ず反映）
各画面仕様で一貫して次を満たす。`要確認` は残してよいが、未定義のまま実装方針を断定しない。

1. **操作×API 対応表を必須化**
   - 各画面で少なくとも次の操作を行単位で列挙する: 初期表示、一覧/検索、詳細表示、作成、更新、承認/却下、状態操作、非同期結果取得
   - 各行に `UC / API / DTO / 画面状態` を対応付ける

2. **参照系 API（一覧 GET / 詳細 GET）を明示**
   - 一覧取得はページング、ソート、フィルタ（期間、会員ID、状態等）を明記する
   - 詳細取得は主キー、取得タイミング、再取得条件（更新後、エラー復帰後）を明記する
   - 参照系 API が `application` に無い場合は「要確認」ではなく `application 追記候補` として書き出す

3. **DTO/ID/型の一貫性を固定**
   - 画面項目、`application` の DTO、`論理データモデル.md` の属性の対応を 1 対 1 で確認する
   - ID 形式（number/string、接頭辞有無）、日時形式、金額の丸め・表示形式を画面内で統一する

4. **非同期・長時間処理の UI 方針を固定**
   - 決済や一括処理などは、POST 後の状態反映方法（ポーリング、再取得、イベント連携）を明記する
   - 待機中、タイムアウト、再試行、二重送信防止を `画面状態` と `エラー表示` に接続する

5. **部分成功・冪等性・競合を明示**
   - 一括操作（複数行処理）は部分成功時の表示（成功件数、失敗行、再実行対象）を定義する
   - 更新系は冪等性、再送挙動、競合（Version/ETag 等）を `application` 記述に合わせる。未定義なら `要確認` とし、実装を断定しない

6. **権限と監査の粒度を統一**
   - 画面単位ではなく操作単位で「誰が何をできるか」を明記する
   - 401/403/404 の表示差、監査ログ対象操作（作成/更新/承認/失敗対応）を統一形式で記載する

7. **上流不足の扱いを統一**
   - Domain 仕様に不足がある場合は `Domain 追記候補`、Application API に不足がある場合は `Application 追記候補` として分類する
   - UI 仕様で独自に URL、DTO、状態遷移、エラーコードを作らない
   - 不足があっても画面の目的・必要操作・必要データは明記し、どの上流仕様が不足しているかを追跡できるようにする

## 品質チェック（生成前に必ず自己検査）
次を満たすまで出力を確定しない。

- 画面名、UC、BUC、アクター、情報、条件が `関連データ.txt` と一致する
- `画面照会.json` の BUC・アクター・画面の関係と矛盾がない
- 画面項目と `application` の DTO/API、および `論理データモデル.md` に対応漏れがない
- Application 仕様の `レイヤー接続チェック` で、当該画面の UC / API / Domain サービス対応が確認できる
- Application 仕様の `UI 連携契約` に、当該画面の操作が列挙されている
- **GUI 要素と機能要求**が対応し、どの操作がどの API を呼ぶか明確である
- **生成するプログラムが `application` 層 API を呼び出す設計になっている**（直接ドメイン呼び出しなし）
- 一覧 GET / 詳細 GET / 更新 API の不足は `application 追記候補` として列挙され、推測実装を確定していない
- API 契約（メソッド、パス、主要 DTO、エラーコード）と画面操作の対応が明確である
- DTO 型、ID 形式、日時/金額フォーマットが画面全体で一貫している
- 非同期処理の再同期方法（ポーリング/再取得等）と画面状態遷移が定義されている
- 一括処理の部分成功、冪等性、競合時の扱いが明示されている
- 権限制御（操作単位）と監査ログ対象操作が定義されている
- **非機能要求**に根拠がある（根拠がないものは「要確認」）
- 推測で確定した事項がない（不明は「要確認」）

## 出力フォーマット制約
- すべて Markdown
- 表と箇条書きを併用して可読性を確保
- 操作フローは Mermaid（`flowchart TD`）で記述可
- 断定できない内容は「要確認」と明示し、推測で確定しない

# 実行

次の順で実行する。

1. **対象画面集合**を確定する（全体実行なら「画面照会」と `#edge UC 画面` の和集合。`TARGET_BUC` 付きなら当該 application の「利用画面」表のみ）。
2. 画面の**日英命名対応表**をチャット応答に**短く**表示する。
3. 対象画面ごとに `3_RDRASdd/ui/<BUCEnglishName>/<ScreenEnglishName>.md` を**作成・上書き**する。
4. ファイル作成後、**実際に出力したファイルパス一覧**をチャット応答に表示する。
5. 出力件数が対象件数と一致しない場合、**完了を宣言せず**、不足画面名と不足理由を表示する。

**禁止**

- 「これから生成します」で終了すること。
- 命名表だけを表示して終了すること。
- Markdown 本文をチャットに表示しただけで、**ファイルを書かずに**完了扱いすること。
- 対象外の `RDRA_Knowledge/` 配下へ出力すること。
- 中間ファイルのみをディスクに書くこと（`_` 始まりの便宜ファイルは作らない）。
