# 32_Create_Application

## 役割
あなたは RDRA 定義を読み解き、実装 AI がそのまま利用できる **アプリケーションレイヤー仕様書** を生成するアーキテクトです。
出力は必ず Markdown で作成し、BUC 単位で分割します。

## 入力ファイル
- `1_RDRA/if/関連データ.txt`
- `2_RDRASpec/ビジネスルール.md`
- `2_RDRASpec/論理データモデル.md`
- `2_RDRASpec/画面照会.json`
- `3_RDRASdd/domain/<ContextEnglishName>.md`（該当 UC が利用する情報・状態を含むドメイン仕様書）

## 関連データ.txt の読み方（必須）
- `#child A B X@@Y//...` : A と B の親子関係。`X@@Y` は `親@@子`。
- `#edge A B X@@Y//...` : A と B の関連。
- `#arrow A B X@@Y//...` : 状態遷移や前後関係など方向性のある関連。
- `@@` : 左右オブジェクトの区切り。
- `//` : 複数ペアの区切り。

## 生成対象
`#child 業務 BUC`、`#child BUC アクティビティ`、`#edge アクティビティ UC` を起点に、BUC ごとに 1 ファイルを生成する。

## Domain / UI との接続方針（必須）
Application 仕様は Domain と UI をつなぐ契約である。各 UC について、次の鎖を必ず確認してから API を定義する。

**BUC → アクティビティ → UC → 情報/状態/条件 → Context → Domain サービス → Application API → 画面操作**

- `#edge UC 情報` から利用する情報を特定し、その情報を持つ `3_RDRASdd/domain/<ContextEnglishName>.md` を参照する。
- `#arrow 状態 UC` / `#arrow UC 状態` から UC の事前状態・事後状態を特定し、Domain の状態モデルと矛盾しないことを確認する。
- Domain 仕様の `Application 連携契約` にあるサービス名、メソッド名、DTO、戻り値、エラー分類を優先して利用する。
- Domain に存在しないサービス名・DTO 名・状態遷移を Application 側で新規に確定しない。不足は `Domain 追記候補` として列挙する。
- UI の初期表示・一覧・詳細・検索・状態再取得に必要な API が UC の主操作と別に必要な場合、`参照系 API` として Application 側に定義する。

## ファイル名規則
- 出力先: `3_RDRASdd/application/<BUCEnglishName>.md`
- `BUCEnglishName` は日本語 BUC 名から英語 PascalCase 名を生成する。
- 命名規則: 名詞句、略語なし、一貫した語彙を使う。

## 出力ルール（必須・厳守）
- 1 回の実行で **対象 BUC 全件** の本体仕様書ファイル（`3_RDRASdd/application/<BUCEnglishName>.md`）を出力する。
- 中間ファイル（命名対応表、テンプレート、ロードマップ、進捗マッピング、`_BUC_Mapping.md` / `_ApplicationSpecTemplate.md` 等）を別ファイルとして作成しない。BUC の日英命名対応表はチャット応答の冒頭にのみ記載し、ディスクには書き込まない。
- "Stage" / "Week" / "次回生成予定" などの段階分割・先送りを行わない。"— 生成予定" の状態を残さない。
- 本体ファイルの英語名（PascalCase）と件数を冒頭で確定させ、`関連データ.txt` の `#child 業務 BUC` に現れる **ユニークな BUC（日本語名、右辺）** の件数と一致するまで完了を宣言しない。
- 出力先以外（特に `RDRA_Knowledge/` 配下）には書き込まない。

## 対象件数の確定基準
- `関連データ.txt` の `#child 業務 BUC` 行の各ペア `業務名@@BUC日本語名` について、**BUC 日本語名（`@@` の右側）のユニーク集合** の要素数が、生成する `<BUCEnglishName>.md` のファイル数と一致すること。

## 出力順序（必須）
1. まず BUC の日英命名対応表を出力する。
2. 命名表を提示した後、各 BUC 仕様書を順に出力する。

## 各アプリケーション仕様書の章立て（詳細）
各ファイルは次の順序で作成する。

1. 概要
   - BUC 日本語名
   - BUC 英語名
   - 所属業務
   - 目的（2-4行）

2. アクターと画面
   - `画面照会.json` の `businesses[].BUCs[]` から抽出
   - アクターごとの利用画面を整理
   - `#edge 画面 アクター` と矛盾がないか確認

3. ユースケース一覧
   - `#child BUC アクティビティ` からアクティビティを抽出
   - `#edge アクティビティ UC` から、アクティビティ配下の UC を漏れなく列挙
   - 必要に応じて業務順に並べる
   - 主要フロー/代替フローを分ける

4. UC 詳細（UCごとに同一フォーマット）
   - 英語名
   - 日本語名
   - 目的
   - 事前条件（`#arrow 状態 UC`）
   - 事後条件（`#arrow UC 状態`）
   - 入力（`#edge UC 画面` と `画面照会.json` の fields）
   - 参照/更新データ（`#edge UC 情報`）
   - 適用条件・ルール（`#edge UC 条件` + `ビジネスルール.md`）
   - タイマー/イベント（`#edge UC タイマー`、`#edge UC イベント`）
   - 例外系（状態不整合、入力不正、外部連携失敗）

5. API エンドポイント
   - UC 単位で REST ライクに設計
   - 日本語名
   - HTTP メソッド、パス、目的、リクエスト、レスポンス、主なエラー
   - DTO は `論理データモデル.md` の属性に基づき定義
   - UI の画面ロードに必要な一覧 GET / 詳細 GET / 検索 GET / 状態確認 GET を、更新系 API とは分けて定義する
   - 一覧 GET はページング、ソート、フィルタ、レスポンス件数、空結果の扱いを明記する
   - 詳細 GET は主キー、取得タイミング、戻り値 DTO、404/403 の扱いを明記する
   - 非同期処理・外部連携を伴う UC は、POST 後の状態確認 API または結果取得 API を定義する
   - API が定義できない場合は推測で補わず、`Application API 未定義事項` として用途、必要画面、必要 DTO、上流の不足理由を列挙する

6. 画面バインディング
   - 画面フィールドと DTO / Entity 属性の対応表
   - 入力必須、編集可否、表示専用の区分
   - 検証ルール（形式、桁、列挙値）を明記

7. ドメイン層呼び出し契約
   - 利用するドメインサービス
     - コンテキスト、サービス名
   - トランザクション境界
   - 冪等性が必要な UC は明示
   - Domain 仕様のサービス名、入力、戻り値
   - Domain のエラー分類を Application API の HTTP ステータス、エラーコード、ユーザー向け概要へ変換する
   - 複数 Context をまたぐ場合は、どの Context を更新し、どの Context は参照のみかを明記する

8. 外部連携
   - `#edge イベント 外部システム` から連携先を抽出
   - 送受信イベント、再試行方針、失敗時の補償

9. 実装 AI 向け指示
   - コントローラ/ユースケース/DTO の疑似シグネチャ
   - I/O 契約を厳密化
   - ログ/監査項目（誰が・いつ・何を）

10. UI 連携契約
   - `画面照会.json` および `#edge UC 画面` から、各 API を利用する画面を列挙する
   - 画面操作ごとに、初期表示、一覧/検索、詳細表示、登録、更新、承認/却下、状態操作、非同期結果取得のどれに該当するかを明記する
   - UI が直接 Domain を呼ばず、この Application API を呼べば完結することを確認する
   - 画面に必要だが API が未定義の操作は、`UI から見た API 不足` として列挙する

## レイヤー接続チェック（生成前必須）
各 BUC 仕様書の末尾に、次の対応表を出力する。

| UC | 利用情報 | Context | Domain サービス/メソッド | Application API | 利用画面 | 状態遷移 | 不足・要確認 |
|----|----------|---------|--------------------------|-----------------|----------|----------|--------------|

- 1 つの UC が複数 API を持つ場合は API ごとに行を分ける。
- 画面初期表示や一覧 GET など、状態遷移を伴わない参照 API も行に含める。
- `不足・要確認` には、Domain 追記候補、Application API 未定義事項、UI から見た API 不足を分類して記載する。

## 品質チェック（生成前に必ず自己検査）
次を満たすまで出力を確定しない。
- BUC、UC、画面、情報、条件、状態が `関連データ.txt` と一致する
- UC の事前条件と事後条件に矛盾がない
- 画面項目と DTO 項目の対応漏れがない
- API 入出力のキーが `論理データモデル.md` と整合する
- ビジネスルールの適用箇所が `ビジネスルール.md` と整合する
- UC から利用情報を経由して Context / Domain 仕様へ到達できる
- Application のドメイン層呼び出し契約が Domain の `Application 連携契約` と一致する
- Domain の状態遷移と Application の事前条件・事後条件が一致する
- UI の初期表示、一覧、詳細、検索、非同期結果取得に必要な参照 API が定義されている
- Domain エラー分類が Application API のエラーコード/HTTP ステータス/UI 概要へ変換されている
- レイヤー接続チェック表に空欄や推測確定が残っていない（不明点は分類して `要確認`）

## 出力フォーマット制約
- すべて Markdown
- 表と箇条書きを併用して可読性を確保
- 必要なフロー図は Mermaid（`flowchart TD`）で記述可
- 断定できない内容は「要確認」と明示し、推測で確定しない

# 実行

このプロンプト 1 回の応答で、上記対象集合の本体仕様書を全件、指定パスに出力してください。中間ファイルや段階生成は行わないでください。
