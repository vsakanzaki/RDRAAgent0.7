# 31_Create_Domain

## 役割
あなたは RDRA 定義を読み解き、実装 AI がそのまま利用できる **ドメインレイヤー仕様書** を生成するアーキテクトです。
出力は必ず Markdown で作成し、コンテキスト単位で分割します。

## 入力ファイル
- `1_RDRA/if/関連データ.txt`
- `2_RDRASpec/ビジネスルール.md`
- `2_RDRASpec/論理データモデル.md`
- `2_RDRASpec/画面照会.json`

## 関連データ.txt の読み方（必須）
- `#child A B X@@Y//...` : A と B の親子関係。`X@@Y` は `親@@子`。
- `#edge A B X@@Y//...` : A と B の関連。
- `#arrow A B X@@Y//...` : 状態遷移や前後関係など方向性のある関連。
- `@@` : 左右オブジェクトの区切り。
- `//` : 複数ペアの区切り。

## 生成対象
`関連データ.txt` の `#child コンテキスト 状態モデル` と `#child コンテキスト 情報` を起点にコンテキストごとに 1 ファイル生成する。

注意:
- `条件コンテキスト` は独立ファイルを作らない。
- 条件は `#edge 条件 状態モデル` で紐づく状態モデルの所属コンテキストへ統合する。

## ファイル名規則
- 出力先: `3_RDRASdd/domain/<ContextEnglishName>.md`
- `ContextEnglishName` は日本語名から英語 PascalCase 名を生成する。
- 命名規則: 名詞句、略語なし、一貫した語彙を使う。

## 出力ルール（必須・厳守）
- 1 回の実行で **対象コンテキスト全件** の本体仕様書ファイル（`3_RDRASdd/domain/<ContextEnglishName>.md`）を出力する。
- 中間ファイル（命名対応表、テンプレート、ロードマップ、進捗マッピング、`_ContextMapping.md` / `_DomainSpecTemplate.md` 等）を別ファイルとして作成しない。日英命名対応表はチャット応答の冒頭にのみ記載し、ディスクには書き込まない。
- "Stage" / "Week" / "次回生成予定" などの段階分割・先送りを行わない。"— 生成予定" の状態を残さない。
- 本体ファイルの英語名（PascalCase）と件数を冒頭で確定させ、`関連データ.txt` の対象コンテキスト件数と一致するまで完了を宣言しない。
- 出力先以外（特に `RDRA_Knowledge/` 配下）には書き込まない。

## 対象件数の確定基準
- `関連データ.txt` の `#child コンテキスト 状態モデル` と `#child コンテキスト 情報` の各ペア（`コンテキスト名@@…`）の **左辺（コンテキスト名）をユニオンした集合** の要素数が、生成する `<ContextEnglishName>.md` のファイル数と一致すること。
- `条件コンテキスト` は独立ファイルを作らない（プロンプト「生成対象」の注意どおり）。

## 出力順序（必須）
1. まず日英命名対応表を出力する。
2. ユーザー確認可能な形で命名表を示した後、各コンテキスト仕様書を順に出力する。

## 各ドメイン仕様書の章立て（詳細）
各ファイルは次の順序で作成する。

1. 概要
   - コンテキスト日本語名
   - コンテキスト英語名(UpperPascalCase)
   - 目的（2-4行）

2. エンティティ定義
   - `論理データモデル.md` から該当エンティティを抽出
   - 属性: 項目名、型、isKey、説明、主キー/外部キー
   - 他コンテキスト参照があれば明記

3. Value Objects / 列挙
   - バリエーション（`#edge 情報 バリエーション`、`#attribute バリエーション 値`）
   - 状態値（該当状態モデルの状態一覧）

4. 状態モデル
   - `#arrow 状態 状態` と `#stereotype 状態 Stereotype` から状態遷移を整理
   - Mermaid で state diagram を記述
   - 遷移トリガーに該当する UC があれば併記（`#arrow 状態 UC` / `#arrow UC 状態`）

5. ビジネスルール
   - `#edge 条件 状態モデル` / `#edge 条件 バリエーション` で条件を抽出
   - `ビジネスルール.md` の該当ルールを要約
   - ルールごとに「適用タイミング」「対象エンティティ」「違反時の扱い」を明記

6. 不変条件と整合性制約
   - 主キー一意性、外部キー整合、状態と属性の整合
   - バリエーションとの組み合わせ制約
   - 監査上必要な制約（あれば）

7. ドメインサービス
   - 状態遷移を伴う操作を UC から抽出
   - 情報のCRUDを出力する
     - 情報に親子関があるものは一つのトランザクション処理とする
       - 親子関係例：親/受注、子/受注明細
   - サービス毎の詳細
     - サービス名、責務、入力、戻り値
     - 変更対象状態、利用するビジネスルール
     - 処理説明

8. コンテキスト境界と依存
   - `#edge 情報 情報` から他情報との関係を整理
   - どの情報を参照のみ/更新可能かを明示

9. 実装 AI 向け指示
   - 言語非依存の疑似シグネチャ（メソッド名、入力 DTO、戻り値）
   - トランザクション境界
   - バリデーション順序
   - エラー分類（業務エラー/整合性エラー/外部連携エラー）

10. Application 連携契約
   - `3_RDRASdd/application` が参照できるように、ドメインサービスごとに次を表で明記する
     - サービス名、メソッド名、入力 DTO、戻り値 DTO / イベント、変更対象エンティティ、変更対象状態、発生し得るエラー分類
   - 情報の CRUD を出力した場合は、各 CRUD がどのエンティティ/集約を対象にし、どの状態・不変条件を検証するかを明記する
   - `#edge UC 情報` / `#arrow 状態 UC` / `#arrow UC 状態` から、どの UC がこのドメイン契約を利用し得るかを `利用候補 UC` として列挙する
   - Application 層で公開 API 化すべき参照操作（一覧、詳細、検索、状態確認）がある場合は、ドメインとして必要な参照条件と戻り値項目を明記する
   - 断定できない UC 対応は `要確認` とし、存在しないサービス名・DTO 名を推測で確定しない

## 品質チェック（生成前に必ず自己検査）
次を満たすまで出力を確定しない。
- 参照した全オブジェクト名が `関連データ.txt` に存在する
- 各状態遷移に矛盾がない（孤立状態なし）
- 条件・情報・状態モデルの結びつきが `#edge` / `#arrow` と一致する
- エンティティ属性が `論理データモデル.md` と一致する
- ルール記述が `ビジネスルール.md` の意図を損なわない
- Application 層が参照できるサービス契約（サービス名、メソッド名、DTO、戻り値、状態変更、エラー分類）が明示されている
- 状態遷移を伴う UC とドメインサービスの対応が `#arrow 状態 UC` / `#arrow UC 状態` と矛盾していない

## 出力フォーマット制約
- すべて Markdown
- 表と箇条書きを併用して可読性を確保
- Mermaid は `stateDiagram-v2` を使用
- 断定できない内容は「要確認」と明示し、推測で確定しない

# 実行

このプロンプト 1 回の応答で、上記対象集合の本体仕様書を全件、指定パスに出力してください。中間ファイルや段階生成は行わないでください。
