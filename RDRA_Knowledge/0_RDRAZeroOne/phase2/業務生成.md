# phase2：業務
- 「0_RDRAZeroOne/phase1/初期要望分析.md」を読み初期の要望を理解する
- 「0_RDRAZeroOne/phase1/業務.tsv」を読み初期要望の業務を理解する
- 「0_RDRAZeroOne/phase1/ビジネスポリシー.tsv」を読み初期要望のビジネスポリシーを理解する
- 「0_RDRAZeroOne/phase1/状態.tsv」を読み初期要望の状態モデルを理解する

# 背景理解
- 「RDRAKnowledge/1_RDRA/RDRA.md」を読込みRDRAの構造を理解してください
- phase1のアクターと業務は初期要望から個別に洗い出している
- 「phase1/業務」をベースに「phase2/業務」を作成する
  
## 業務の洗出し
- 「phase1/業務」の項目をそのまま「phase2/業務」に出力する
- 「phase1/業務」の説明を参考に「業務/BUC」をブレークダウンしアクティビティとUCを追加する
- BUCの複数アクティビティが業務フローを実現する
- アクティビティは仕事の単位となる
- UCは仕事でコンピュータを使う仕事の単位となる
- 「初期要望分析.md」に業務フローの要素になるものがあれば、アクティビティとして追記する
- 「初期要望分析.md」にシステムの機能について記述があれば、UCとして追記する
- 「状態.tsv」の状態モデル内の状態の遷移はUCによって遷移される
- 状態モデルに関わるBUCを識別し、状態を遷移させるUCは業務内のUCに対応する

## 出力ルール
- 出力には必ずヘッダー（カラム名）行を出力する
- 前後に「```tsv」「```」を付ける
- 「業務」以外の説明を出力しない

### 「業務」のフォーマット
- システム化対象の業務を定義する
- ７カラムのタブ区切りで出力してください
```tsv
「業務、BUC、価値受益アクター、価値提供アクター、アクティビティ、UC、説明」
- 業務：業務を分類する名前　例：「販売管理」「在庫管理」など
- BUC：洗い出した「BUC」の名前。
- 価値受益アクター：BUCから価値を得る一つのアクター
- 価値提供アクター：BUCの価値を提供するために関わる複数のアクター
- アクティビティ：仕事の単位となる
- UC：ユースケース（UC）アクティビティで使うシステムを表す
- 説明：BUCの内容を説明
```
