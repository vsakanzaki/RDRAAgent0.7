# phase1：業務
- 「初期要望.txt」から業務、業務フローの単位を洗い出し「0_RDRAZeroOne/phase1/業務.tsv」を出力する
- 業務の洗出しにはアクターが重要な役割を担うので、アクターを洗出す
- 業務/BUCを洗い出した後に業務/BUCとアクターの関係も洗い出す

# 背景理解
- 「RDRA_Knowledge/1_RDRA/RDRA.md」を読込み理解してください
- 初期要望は実現したいビジネスとそれを実現するシステムについて記述している
- ここではRDRAモデルの中の「業務」を洗い出す

# [洗出しRule]
## アクターの洗い出し方法
- 初期要望に「アクター」の記述がある場合は、そのアクターを洗い出す
- さらに以下のルールで「アクター」を追加する
  - アクターは役割を表す名前となる
  - 価値受益アクター、価値提供アクターは"管理者"や"担当者"、"事務職"などの単純な名前ではなく、役割が分かる名前にする。
  - 組織名や会社名、支店(事業所)などを想起する名前にしない
  - アクターが社内のアクターか社外のアクターを識別する
  - 一つのアクターに多数の役割を担わせない
    - 複数の役割があるアクターは分割する
  - システムや外部機関などの人の役割ではないものはアクターにしない
  - 洗い出すアクターの数は業務の数の倍程度ある

### 【重要】アクター分割の定量ルール
- **一つのアクターが価値提供アクターとなる業務（業務カラムのユニーク値）は1つだけとする**（＝主担当業務は1つ）
- 上記を超える場合は、担当する業務ごとにアクターを分割する
- 「事務担当者」「管理者」「担当者」のような汎用的な名前は使わない。必ず担当業務の内容が分かる名前にする

### アクターの例
- 顧客アクターは販売業務では価値受益アクターになる
  - 顧客アクターは社外のアクターなので主担当業務も関連業務もない
- 営業アクターは販売業務では価値提供アクターになり、販売業務が主担当業務になる
- 営業事務アクターは販売業務では価値提供アクターになり、販売業務が主担当業務になる
- 営業事務アクターは物流業務では価値受益アクターで、物流業務が関連業務になる
  - 営業事務アクターは物流業務に対して出荷指示を出すので価値受益アクターになる

### アクター分割の具体例
- 悪い例：「事務担当者」が会員管理、スタッフ管理、費用計算、請求を全て担当
- 良い例：業務ごとに役割を分割する
  - 「会員管理担当者」→ 会員管理業務の価値提供アクター
  - 「スタッフ管理担当者」→ スタッフ管理業務の価値提供アクター  
  - 「経理担当者」→ サービス費用業務の価値提供アクター
  - 「実績管理担当者」→ サービス実施業務の記録確認の価値提供アクター

## 業務/BUCの構造
- 業務は「販売管理業務」「在庫管理業務」などのように企業の中の大きな管理単位となる
- 業務は１つ以上のBUCを含み、階層構造になる
- BUCは業務フローの単位となる

## アクターとBUCの関係
- 初期要望から**主体**となる会社もしくは組織を明らかにする
  - **主体**が会社の場合は社外のアクターが優先度の高いアクターになる
    - 社外のアクターがいる場合は**主体**は会社になる
  - **主体**が組織の場合はバリューストリームの上流にある組織を優先する
- 優先度の高いBUC
  - 社外のアクターが関わるBUCは優先度が高い
  - 売上に関わるBUCは優先度が高い
  - バリューストリームの上流にあるBUCが優先度の高いBUCになる
- 会社に対する取引の入り口がビジネスイベントになる
- ビジネスイベントから始まるBUCの価値受益アクターは社外のアクターになる

### アクターとBUCの関係
- 社内起点のBUCの価値受益アクターは社内アクターになる
- 社内のアクターが開始するBUCは、そのアクターが価値受益アクターになる
- 社内リソースを扱うBUCの価値受益アクターは社内のアクターになる

### BUC視点でのアクターとの関係
- BUCは価値受益アクターに価値を提供する
  - BUCが価値を提供するのは一人の価値受益アクターである
  - BUCには必ず一人の価値を得るアクターがいる
- BUCを開始（最初のアクティビティ/UC）するアクターは価値受益アクターになる
  - 業務/BUCは価値受益アクターによって開始される
  - 社内のアクターは異なるユースケースとの関りにおいて価値提供アクターと価値受益アクターになれる

### アクター視点でのBUCとの関係
- アクターは一つの主担当業務をもつ。主担当業務は一つだけである
- アクターは主担当業務以外に複数の関連業務に関わることができる
- 一つのアクターに複数のロールを当てはめない
- 役割が違う場合は別のアクターにする
- 価値受益アクターになるアクターは、そのアクターにとっては関連業務になる
- 価値提供アクターになるアクターは、そのアクターにとっては主担当業務になる
- **主担当業務が複数になるアクターは存在してはならない。そのようなアクターが発生した場合は、業務ごとに別のアクター名に分割すること**
- 社長、事業部長、支社長、事業所長などの管理者は、その職責でしかできないBUCに当てはめる
- **BUCの価値受益アクターと価値提供アクターが同じアクターになることは絶対に禁止する。同一人物が両方の役割を担う場合でも、異なるアクター名で表現する**

## 業務とBUCの洗出し方法
- 初期要望からビジネスとして実現すべき業務とその中の業務フローとなるBUCを洗い出す
- 初期要望に記述されていない業務フロー(BUC)も生成する
- BUCは情報を通じて連携する
- BUCのつながりがバリューストリームを構成する
  - BUCの洗出しは優先度の高いBUCから始まり、連携するBUCでバリューストリームを形作る
- 洗い出すBUCはバリューストリームに関わるBUCを洗い出す
- バリューストリームを補完するBUCも洗い出す
  - リソースの保全、保守するBUCはバリューストリームを関するBUCになる
- BUCには社内リソースを扱うBUCがある
  - 社内リソースとは在庫や設備、機器などを指す
    - 例：在庫管理は社内リソースを扱うBUCである
  - 販売業務や調達業務は社外のアクター、社外のシステムに関わる

## 利用リソース
- BUCの業務フローの中で必ず使うリソースを洗い出す
- リソースとは機器、設備、材料などを指す
- BUCを実行するために無くてはならないリソースを洗い出す
  - 例：製造の業務フローにはリソースとなる製造設備、製造機器が必要
  - 例：販売フローにはリソースは必要ない
  - 例：訪問介護実施の業務フローには医療機器、医療器材が必要
- 洗い出すリソースは管理対象の資産に限定する
  - 個々の機器や材料などの細かいリソースは洗い出さない

## 情報の洗出し
- 初期要望からビジネスとして管理すべき情報を洗い出す。
- 情報は必ずユニークなIDを持ち、一意に識別できる
- ビジネスは情報に状態を保持する
- 「初期要望.txt」にデータに関わる記述がある場合は、情報として洗い出す

### 情報とBUCの関係
- BUCは業務フローの単位となり、その中で情報を操作する
- BUCの中で扱う情報は取引を記録する
- ビジネスを行う上で必要な情報をBUCに結びつける
- BUCは情報を経由して他のBUCとつながり、ビジネスを進める

## 外部システム
- 初期要望からシステムに必要な外部のサービスを洗い出す。
- 外部のサービスがシステムの場合、外部システムになる
- 外部システムには役割があり、その役割に応じてBUCとつながる

### BUCと外部システムの関係 
- 外部システムの役割において、BUCの業務フローと関わるものを洗い出し、BUCに結びつける

## 実行ルール
- 初期要望から業務とBUCが導出できないときはヘッダーだけをファイルに出力する
- 出力内容について判断に迷った場合は可能性の一番高いものを選択する
-「業務」出力フォーマットで出力する

# [洗出し実行]

## 実行順序
- Step1:アクターを一旦洗い出す（Step10で見直す）
- Step2:外部システムを洗い出す
- Step3:業務/BUCを洗い出す
- Step4:リソースを洗い出す
- Step5:情報を洗い出す
- Step6:業務/BUCに価値受益アクターと価値提供アクターを結びつける
- Step7:業務/BUCに利用リソースを結びつける
- Step8:業務/BUCに情報を結びつける
- Step9:業務/BUCに外部システムを結びつける
- Step10:アクター集中度を検証し、違反があればアクターを分割して再構成する

### アクター集中度検証
以下のチェックを行い、違反があればアクターを分割してStep6からやり直す：
1. 各アクターが「価値提供アクター」として登場する「業務」のユニーク数を数える → 2以上のアクターは分割が必要
2. 各アクターが「価値受益アクター」として登場するBUCの数を数える → 3以上のアクターは分割を検討する
3. 同一BUCで価値受益アクターと価値提供アクターが同じアクター名になっていないか確認する → 一致している場合はアクター名を変更する

## 出力ルール
- 出力ファイル：「0_RDRAZeroOne/phase1/業務.tsv」
- 出力には必ずヘッダー（カラム名）行を出力する
- 空フィールドにするときは、タブ区切りを省略しない
- プログラムは絶対に生成しない。

### 「業務」出力フォーマット
出力形式：TSV（タブ区切り）
カラム構成：
1列目: 業務：業務を分類する名前　例：「販売業務」「在庫業務」など、名前の最後には「～業務」を付ける
2列目: BUC：洗い出した「BUC」の名前。名前の後ろに「～フロー」をつける
3列目: 価値受益アクター：BUCから価値を得る一つのアクター
4列目: 価値提供アクター：BUCの価値を提供するために関わる複数のアクター。複数ある場合は「、」区切りで設定する
5列目: 利用リソース：BUCが扱うリソースを設定する。複数ある場合は「、」区切りで設定する。該当なしの場合は空フィールドにする
6列目: 情報：BUCが主に扱う情報を設定する。複数ある場合は「、」区切りで設定する
7列目: 外部システム：BUCが関わる外部システムを設定する。複数ある場合は「、」区切りで設定する。該当なしの場合は空フィールドにする
8列目: システム化有無：「〇」を設定する
9列目: 説明：BUCの内容を説明する
- ヘッダー行を含め、各フィールドはタブ文字で区切ってください。
- 空フィールドの場合は「、」だけを出力しない

## 出力チェックリスト
- [ ] 初期要望に具体的に記述されているアクターが価値受益アクターか価値提供アクターに必ず含まれていなければならない
- [ ] 各行が9カラム（8個のタブ区切り）であることを確認する
- [ ] 各アクターが価値提供アクターとして関わる業務（1列目のユニーク値）は1つだけであること
- [ ] 同一BUCの価値受益アクターと価値提供アクターに同じアクター名が含まれていないこと
- [ ] 「担当者」「事務担当者」「管理者」のような汎用的なアクター名が使われていないこと
