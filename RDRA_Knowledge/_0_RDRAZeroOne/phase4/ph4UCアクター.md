# phase4：UC-アクター
- 「0_RDRAZeroOne/phase3/ph3BUC.tsv」を読み込み業務/BUC/アクティビティ/UCの構造とUCにつながる情報、アクター、状態モデル、状態の関係を理解し、UCの役割を深く理解する
- 「ph3BUC.tsv」の記述は上記のファイルのパスを指す
- 「初期要望.txt」を読み人が扱うインターフェースになる記述を把握する
- 「ph3BUC.tsv」のUCとアクターの関係から画面を生成し「ph4UCアクター.tsv」を出力する

# 背景理解
- 「RDRA_Knowledge/_1_RDRA/RDRA.md」を読込みRDRAの構造を理解してください
- アクターは画面を経由してユースケース(UC)を操作する
- アクターは役割に応じてUCに関わる
- 画面に対する画面要求は仕様作成に利用する

## UCに関わるアクターから、画面、画面要求を生成する
- 「ph3BUC.tsv」からUCが行うことを理解し、関わるアクターを把握する
- 初期要望.txtに画面が記述されている場合は、それを名前にする
- 初期要望.txtに画面が記述されていない場合はアクターとUCを連携させる画面を生成する
- 画面に対する画面要求を生成する

### 画面名の命名規則
- 初期要望.txtに画面が記述されている場合は、決して新たな名前を生成しない
- 画面はアクターがUCを操作するためのインターフェースとなる
- 画面名は「UC」が提供する「画面」として適切な名前にする
- 画面名は人とのインターフェースの意図が分かる短いシンプルな名前にする
  - UCとアクターをつなげた単純な名前にはしない、
- 名前の最後に「～画面」を付ける

### 画面要求の設定
- 画面要求は画面を設計するための要求として使われる
- 画面に対する要求はアクターの役割から以下のことを実現する要求を設定する
  - 関わるアクターの役割から行わなくてはいけないことができるための画面設計のための要求を設定する
    - 関わるアクターが知っていること、判断できることを行うための要求を設定する
    - 関わるアクターにとって使いやすい要求になっていること

## 出力ルール
- 出力には必ずヘッダー（カラム名）行を出力する
- 空フィールドにするときは、タブ区切りを省略しない
- プログラムは絶対に生成しない。

### 「UCアクター」のフォーマット
出力先：0_RDRAZeroOne/phase4/ph4UCアクター.tsv
出力形式：TSV（タブ区切り）
カラム構成：
1列目: UC：「ph3BUC.tsv」のUCをセットする
2列目: 画面：アクターが操作する画面名をセットする
3列目: アクター：「ph3BUC.tsv」のアクター名をセットする
4列目: 画面要求：画面を開発するときの要求を設定する
- ヘッダー行を含め、各フィールドはタブ文字で区切ってください
