# phase3：業務/BUC/アクティビティにつながるUCを洗い出す
- 「0_RDRAZeroOne/phase1/ph1業務.tsv」を読み初期要望の業務を理解する
- 「0_RDRAZeroOne/phase1/ph1ビジネスポリシー.tsv」を読み初期要望のビジネスポリシーを理解する
- 「0_RDRAZeroOne/phase1/ph1ビジネスパラメータ.tsv」を読みパラメータによってビジネスの多様性を対応することを理解する
- 「0_RDRAZeroOne/phase1/ph1要求.tsv」を読み機能要求/非機能要求を理解する
- 「0_RDRAZeroOne/phase2/ph2状態.tsv」を読み必要な状態モデルと、管理する状態を理解する
- 「0_RDRAZeroOne/phase2/ph2条件.tsv」を読み必要なビジネスルールである条件を理解する
- 「0_RDRAZeroOne/phase2/ph2アクティビティ.tsv」を読み業務/BUC毎のアクティビティの役割を理解する
- 「ph1業務.tsv」「ph1要求.tsv」「ph1ビジネスポリシー.tsv」「ph1ビジネスパラメータ.tsv」「ph2状態.tsv」「ph2条件.tsv」「ph2アクティビティ.tsv」の記述は上記のファイルのパスを指す
- 上記ファイルから業務を再整理し「ph3BUC.tsv」を出力する

## 背景理解
- 「RDRA_Knowledge/_1_RDRA/RDRA.md」を読込みRDRAの構造を理解してください
- 「ph1業務.tsv」のシステム化有無が「〇」のものはコンピューターが関わるBUCになるのでアクティビティにUCを結びつける
- 「ph1業務.tsv」のシステム化有無が「〇」以外はコンピューターが関わらないのでUCを洗い出さない
- UCはアクティビティの中でコンピュータを使う一塊の作業単位となる

## 「ph2アクティビティ.tsv」から「ph3BUC.tsv」の出力
- 「ph2アクティビティ.tsv」と「ph3BUC.tsv」で同じ項目はそのまま出力する
  - 「業務」「BUC」「アクティビティ」「次」カラムは「ph2アクティビティ.tsv」の値をそのまま引き継ぐ
- 「ph1業務.tsv」のシステム化有無にかかわらず、出力行の列数は常に13カラム固定とする
  - システム化有無が「〇」の場合は、UC〜説明まで必要な値を設定する
  - システム化有無が「〇」以外の場合は、5列目以降も含めて13カラムを維持し、該当しない列は空フィールドにする

### 「ph2アクティビティ.tsv」をベースにアクティビティ毎に以下の処理を行う

#### UCの洗出し（システム化有無が〇の場合）
- アクティビティに関わるUCの洗出しは以下の2パスで行う
  - **第1パス（業務起点の洗出し）**: アクティビティの役割を担うUCを洗出す
  - **第2パス（状態遷移による網羅性チェック）**: 第1パスのUCが状態モデルの全遷移をカバーしているか検証し、不足があればアクティビティを追加する
- 第1パスが主、第2パスは補完である。UCの名前と粒度は業務行為が決める

##### 第1パス
- アクティビティの内容に応じてシステムが行うことをUCとして洗い出す
  - UCの粒度は「アクターがシステムを使って一つのまとまった仕事を完了する単位」とする

##### 第2パス 状態遷移による網羅性チェック
- 「ph2状態.tsv」を読み込み必要な状態遷移を把握する
  - 「ph2状態.tsv」の状態を分割し、そこから状態遷移を導き出す
- 第1パスで洗出したUCが、全状態モデルの中から状態遷移に対応するUCであるかを確認する
  - 全状態モデルについて以下の事を行う
    - 状態遷移はUCが行うので、遷移を行うUCを第1パスで洗い出したUCから探す
- 重要：UCは「コンピュータが行うべきこと」であり、状態遷移はUCを実行した「結果」として発生するものである
  - UC名に状態名や遷移名をそのまま使わない
    - 「会員を休会にする」ではなく「サービス利用を休止する」：業務行為として命名する
    - 状態遷移はUCの結果であり、UC名は「何を行うか（業務行為）」を表す
  - 一つのUCが複数の状態を遷移させることがある
  - 一つの状態遷移が複数のUCの結果として発生することもある
  - 状態モデルは複数のBUCを横断してUCと関わる
    - 第1パスで洗出したUCについて毎回、全状態モデルの状態遷移に対応するかを確認する
- UCには状態モデルが関わらないものもある
  - 照会などの状態を変えないようなUCは状態モデルが関わらない

##### 状態モデルの網羅性チェック例
###### 状態モデルから状態の類推
- 状態モデル「会員状態」　状態「入会済み,活動中,休会中,退会済み」

###### 第1パスで洗出されたUC
UC例：
「入会申し込みを受け付ける」
「会員情報を登録する」

###### チェック：上記UCが状態遷移を網羅しているか？
- 「→活動中」の遷移は？ → 上記UCでカバー済み（入会申し込み→登録で活動中になる）
- 「活動中→休会中」の遷移は？ → 該当UCなし → アクティビティ「サービス休止」とUC「サービス利用を休止する」を追加
- 「休会中→活動中」の遷移は？ → 該当UCなし → アクティビティ「サービス再開」とUC「サービス利用を再開する」を追加
- 「活動中→退会済み」の遷移は？ → 該当UCなし → アクティビティ「退会手続き」とUC「退会手続きを行う」を追加

#### UCの洗出し（システム化有無が〇以外の場合）
~、アクティビティ、次、UC、情報、条件、アクター、外部システム、状態モデル、From状態、To状態、説明
------------------------------------------
~、アクティビティA、↓、[空]、[空]、[空]、[空]、[空]、[空]、[空]、[空]、説明1      

#### UCの洗出し（状態遷移が関わらない場合）
~、アクティビティ、次、UC、情報、条件、アクター、外部システム、状態モデル、From状態、To状態、説明
------------------------------------------
~、アクティビティA、↓、入会申し込みを受け付ける、会員情報、条件A、アクターA、外部システムA、[空]、[空]、[空]、説明1      

#### UCへの関連付け
##### UCと情報の関係づけ
- 情報とは
  - 情報はIDとして一意に管理するentityである
- UCは情報とつながりシステムの整合性を維持する
  - 整合性を維持するために必要な情報を設定する
- UCで行ったことを情報に記録する
  - 例：「出張旅費申請」UCは申請された内容を「出張旅費申請情報」に保持する
- 商品や取引先などのマスターとして参照する情報もUCにつなげる
- 以下のものを情報として洗出さない
  - 分類や区分などを情報にしない、バリエーションとして分類する
  - 人や組織などを情報にしない、アクターとして分類する
  - 状態は情報にしない、状態モデルとして分類する
  - entityのAttributeを情報として洗い出さない

##### UCに外部システムを関連付ける
- UCの中で外部システムと関わるものを探り関連付ける
  - 外部システムの役割を必要とするUCに関連付ける

##### UCに条件を関連付ける
- 「ph2条件.tsv」の全条件を把握し、関係するUCに条件を関係づける
  - 条件の活用場所に対応するUCに条件を関係づける

##### UCに状態モデル、From状態、To状態を関連付ける
- 状態遷移の関わるUCには状態モデル、From状態、To状態を設定する
- 状態モデルはFrom状態→To状態の遷移が発生する場合のみ設定する。照会・参照・記録のみのUCは状態が変化しないため、たとえそのエンティティに状態モデルが存在していても状態モデル列は空にする

###### 状態モデル、From状態、To状態の出力パターン
- ３つのカラム（状態モデル、From状態、To状態）の出力は以下の３パターンに限定する
~、状態モデル、From状態、To状態、説明
--------------------------------
~、状態モデルA、[空]、状態A1、説明A   //状態A1は最初の状態
~、状態モデルA、状態A1、状態A2、説明B //UCが状態A1から状態A2へ状態を遷移させる
~、[空]、[空]、[空]、説明C           //UCは状態を遷移させない

### アクティビティとUCの追加
- 全アクティビティの処理が終わった後に状態モデルの中で状態遷移に対応するUCがない場合は、アクティビティとUCを追加する
  - アクティビティとUCを追加する場合は対応する業務を洗出し、そこにBUC、アクティビティ、UCを追加する
  - 上記のレコードに「UCへの関連付け」を行う

### 「次」カラムの設定ルール
- 「ph2アクティビティ.tsv」の「次」カラムの値をそのまま引き継ぐ
- アクティビティとUCの追加によって新たに追加した行の「次」カラムは以下のルールで設定する
  - 次の行のアクティビティとの連続性がある場合は「次」カラムには「↓」をセットする
    - 連続性がある：このアクティビティが終わらないと次の行のアクティビティが実施できないもの
  - 次の行のアクティビティと連続性がない（並行作業可能）場合は「次」カラムには空フィールドを出力する
    - 連続性がない：このアクティビティと次の行のアクティビティが並行動作可能
  - 次の行のBUCが異なる場合は「次」カラムには空フィールドにする
  - 連続性があるアクティビティが複数行が続くときは「次」カラムに同じものを設定する

## 出力ルール
- 空フィールドにするときは、タブ区切りを省略しない
- 全行は常に12個のタブ区切り（13カラム）で出力する
- 出力には必ずヘッダー（カラム名）行を出力する
- プログラムは絶対に生成しない

### 「BUC」のフォーマット
出力先（固定）：0_RDRAZeroOne/phase3/ph3BUC.tsv
※プロジェクトルート相対、/ 区切り、絶対パス禁止
出力形式：TSV（タブ区切り）
カラム構成：
1列目: 業務：「ph1業務.tsv」の業務をセットする　
2列目: BUC：「ph1業務.tsv」のBUCをセットする
3列目: アクティビティ：洗出した仕事の名前
4列目: 次：次の行のアクティビティにつながる場合は「↓」を設定、次の行のアクティビティと並行動作する場合は空フィールドにする
5列目: UC：ユースケース名。アクティビティでシステムが関わらない場合は空フィールドにする
6列目: 情報：UCが操作する情報、複数ある時は「,」で区切る
7列目：条件：UCが従う条件を設定する。該当なしの時は空フィールドにする
8列目: アクター：UCに関わるアクター。該当なしの時は空フィールドにする
9列目: 外部システム：UCと関わる外部システム、複数ある時は「,」で区切る。該当なしの時は空フィールドにする
10列目: 状態モデル：UCが状態を遷移させるときは状態モデルを設定する。遷移させないとき（From状態、To状態ともに空フィールド）は空フィールドにする
11列目: From状態：UCが遷移可能な状態を表す。遷移に関わらないときとTo状態が開始状態の時は空フィールドにする
12列目: To状態：UCの遷移先となる状態を表す。遷移に関わらないときは空フィールドにする
13列目: 説明：UCの内容を説明　UCがない場合はアクティビティの説明を行う
- ヘッダー行を含め、各フィールドはタブ文字で区切ってください。

#### UC名
- UC名は「XXXをYYYする」形式で作成する
  - XXX：対象　YYY：動詞

## 行生成手順
- 各行は必ず 1列目〜13列目の値を順番に決めてから出力する
- 値がない列は空フィールドを明示的にセットする
- 13個の値がそろった後でのみ、タブ文字で連結して1行にする
- 途中の空フィールド列を理由に列を詰めない
- 説明は常に13列目に置く

## 出力チェックリスト
- [ ] 状態モデルが空なら、10〜12列目はすべて空フィールドである
- [ ] From状態とTo状態が同一状態の行は出力しない（自己遷移の禁止）
- [ ] From状態とTo状態が空フィールドの時は状態モデルも空フィールドになっているか
- [ ] 行ごとに列数が変化していない。全行13カラム存在するか
