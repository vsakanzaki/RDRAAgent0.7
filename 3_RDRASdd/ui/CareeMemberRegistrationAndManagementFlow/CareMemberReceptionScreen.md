# UI仕様書: 介護会員受付画面

## 1. 概要

**画面日本語名**: 介護会員受付画面  
**画面英語名**: CareMemberReceptionScreen  
**所属BUC（日本語）**: 介護会員登録・管理フロー  
**所属BUC（英語）**: CareeMemberRegistrationAndManagementFlow  
**所属業務**: 介護会員管理業務  

**目的**: スケジュール計画担当者が新規の介護会員からの相談・問い合わせ時に基本情報を受け付け、初期登録状態で保持する。受け付けた情報は後段の介護会員登録画面で確認・検証されシステムに登録される。

---

## 2. 利用アクターと役割

| アクター | 役割・責務 | 操作目的 |
|---------|---------|---------|
| スケジュール計画担当者 | 新規会員からの相談・問い合わせを受け付け、会員の基本情報を入力・確認する。スケジュール計画に必要な会員情報の第一次取得を担当 | 新規会員情報の入力確認 |

**アクター別の利用差**: 
- 表示項目: 全て同じ（1アクターのみ）
- 操作可否: 新規入力・送信可能

---

## 3. 関連UC

- **UC-001: 介護会員情報を受け付ける**  
  新規会員からの相談・問い合わせ時に基本情報を受け付ける。入力値をバリデーション後、APIで受付し初期登録状態で保持。  
  参照: `3_RDRASdd/application/CareeMemberRegistrationAndManagementFlow.md` § 4. UC-001

---

## 4. 画面目的

スケジュール計画部門の担当者が、新規会員からの相談・問い合わせ時に基本情報（名前、住所、電話番号、生年月日、施設タイプ）を入力し、システムで初期登録状態として保持する。後続の介護会員登録画面での確認・検証に備えるための入力フェーズ。

---

## 5. レイアウト方針

**セクション構成**:
- **ヘッダー**: 画面タイトル「新規介護会員情報受付」
- **メインセクション**: 基本情報フォーム
  - 会員ID（表示専用、システム自動採番予定）
  - 名前（必須、テキスト入力）
  - 住所（必須、テキスト入力）
  - 電話番号（必須、電話番号形式）
  - 生年月日（必須、日付入力）
  - 施設タイプ（必須、選択肢）
- **フッター**: 送信ボタン、キャンセルボタン

**読み順**: 上から下へ、左から右へ自然な流れ。入力完了後、「受付」または「送信」ボタンで次ステップへ。

**情報密度**: 密度中程度。新規入力フォームのため、入力項目を視認しやすく配置。

**デバイス対応**: デスクトップ・タブレット主体（窓口業務想定）。レスポンシブ対応推奨。

## 8. API利用（UCエンドポイント）

**メソッド/パス**: POST /api/v1/caremembers/accept

**用途**: 新規会員情報を受け付け、初期登録状態で保持

**リクエストボディ**:
```json
{
  "name": "string",
  "address": "string",
  "phoneNumber": "string",
  "birthDate": "string (YYYY-MM-DD)",
  "facilityType": "enum (小規模施設|中規模施設|大規模施設)"
}
```

**レスポンス (201 Created)**:
```json
{
  "careeMemberId": "number",
  "name": "string",
  "address": "string",
  "phoneNumber": "string",
  "birthDate": "string",
  "facilityType": "string",
  "memberStatus": "相談中",
  "careeMemberState": "初期登録",
  "createdAt": "datetime"
}
```

**エラーハンドリング**:
- **400 Bad Request**: 必須項目が不足またはフォーマットが不正  
  → UI概要: 「入力値が不正です。必須項目を確認してください。」エラーメッセージを画面上に表示
- **422 Unprocessable Entity**: バリデーション失敗（電話番号形式等）  
  → UI概要: 「電話番号の形式が正しくありません。」など項目別エラーメッセージを表示

## 9. 画面状態

| 状態 | 説明 | 表示内容 |
|-----|------|--------|
| 初期表示 | 画面初期ロード時 | 空のフォーム。会員IDは非表示またはグレーアウト |
| 入力中 | ユーザーが項目を入力している | 入力値をリアルタイム表示。各項目でバリデーションエラーを表示 |
| 検証エラー | バリデーション失敗時 | エラーメッセージを項目の下に赤色で表示。送信ボタンは無効化 |
| 送信中 | API呼び出し中 | 送信ボタンは「処理中...」表示で無効化。スピナーまたはローディングインジケーター |
| 送信成功 | API呼び出し成功 | 成功メッセージをトーストまたはモーダルで表示。画面を介護会員登録画面へ遷移 |
| 送信失敗 | API呼び出し失敗 | 失敗メッセージを表示。入力値は保持。ユーザーは再送信可能 |

---

## 10. 操作フロー

```
START
  ↓
[画面初期化] → 空のフォーム表示
  ↓
[入力] ← 利用者が各項目を入力・修正
  ↓
[バリデーション（リアルタイム）] → エラー時は項目下にメッセージ表示
  ↓
[送信ボタン押下]
  ├→ [バリデーション失敗] → エラー表示、入力画面に戻る
  └→ [バリデーション成功]
    ↓
  [API: POST /api/v1/caremembers/accept] → リクエスト送信
    ↓
    ├→ [成功 (201)] → 成功メッセージ表示 → 介護会員登録画面へ遷移
    └→ [失敗 (4xx/5xx)] → エラーメッセージ表示、入力画面に戻る
  ↓
[キャンセルボタン] → 確認ダイアログ表示 → 前画面へ戻る
  ↓
END
```

## 13. バリデーション

| 項目 | 検証ルール | タイミング | エラーメッセージ |
|-----|---------|---------|--------------|
| 名前 | 1-100文字、非空 | onBlur, 送信時 | 名前は1~100文字で入力してください |
| 住所 | 1-200文字、非空 | onBlur, 送信時 | 住所は1~200文字で入力してください |
| 電話番号 | 正規表現マッチ（例: `^\d{2,4}-\d{3,4}-\d{4}$` または `^0\d{1,4}-?\d{1,4}-?\d{4}$`） | onChange（フォーマット）, 送信時 | 電話番号の形式が正しくありません。（例: 03-XXXX-XXXX） |
| 生年月日 | YYYY-MM-DD形式、過去日付のみ、100歳未満 | onChange, 送信時 | 生年月日の形式が正しくありません。または年齢が不正です |
| 施設タイプ | 列挙値のみ（小規模施設, 中規模施設, 大規模施設） | 送信時 | 施設タイプを選択してください |

## 14. エラー表示

| エラー種別 | 発生条件 | 表示方法 | 再試行方法 |
|----------|--------|--------|---------|
| 入力検証エラー | バリデーション失敗 | 項目下に赤色テキスト表示 | 入力値修正後、再送信ボタン押下 |
| API呼び出しエラー（400/422） | リクエストボディが不正 | 画面上部にトーストメッセージ表示 | 入力値確認後、再送信 |
| API呼び出しエラー（500以上） | サーバーエラー | 画面上部にトーストメッセージ表示 | 時間経過後、再送信またはサポート連絡 |
| ネットワークエラー | API通信失敗 | 「通信エラーが発生しました」メッセージ表示 | 再送信ボタン押下 |

## 16. 実装AI向け指示

**コンポーネント分割**:
1. フォームコンポーネント（`CareeMemberReceptionForm`）
   - 各入力フィールド、バリデーション、送信処理を管理
   - 状態管理: formData（入力値）, errors（エラー）, isSubmitting（送信中フラグ）

2. API通信コンポーネント
   - `useCareMemberAPI` カスタムフック
   - `POST /api/v1/caremembers/accept` の呼び出し、レスポンス処理、エラーハンドリング

**状態管理の要点**:
- フォーム入力値は `formData` ステート（React等）で管理
- バリデーション結果は `errors` オブジェクトで管理
- API送信中は `isSubmitting` フラグで送信ボタンを無効化
- 成功時は `useNavigate` 等で次画面へ遷移

**API呼び出し順序**:
1. フォーム送信ボタン押下
2. フロント側バリデーション実行
3. バリデーション成功 → isSubmitting=true, 送信ボタン無効化
4. POST /api/v1/caremembers/accept 呼び出し
5. レスポンス受信 → isSubmitting=false
6. 成功 → 成功トースト表示 → 遷移
7. 失敗 → エラートースト表示 → 入力画面に戻す
