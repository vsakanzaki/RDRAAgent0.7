# 関連データに基づくビジネス／システム問合せ回答

`1_RDRA/if/関連データ.txt` を**唯一の事実ソース**として、ビジネス／システムに関する問合せに答える。
回答は**関連データに書かれているオブジェクトとそのつながりから類推できること**に限定し、
推測が必要な箇所はそれと明示する。説明は **わかりやすい日本語の文章**で行う。

## データソース（正）

- 事実: [1_RDRA/if/関連データ.txt](1_RDRA/if/関連データ.txt)
- 行フォーマット: [RDRA_Knowledge/.rdracore/RDRAGraph.md](RDRA_Knowledge/.rdracore/RDRAGraph.md)
- メタモデル（モデル間の規則）: [RDRA_Knowledge/.rdracore/RDRA.md](RDRA_Knowledge/.rdracore/RDRA.md)

## 回答方針（問合せ種別ごと）

問合せを以下のいずれかに分類してから回答する。混在する問合せは複数観点で分けて答える。

### A. ビジネス／業務／仕事に関わる問合せ

- 観点: **業務 → BUC → アクティビティ** の階層をたどる。
- 使用する辺:
  - `#child 業務 BUC` … 業務にどんな BUC が含まれるか
  - `#child BUC アクティビティ` … BUC を構成する仕事の流れ
  - `#edge アクティビティ アクター` / `#edge アクティビティ 外部システム` … 誰が・何と関わって行うか
  - `#arrow アクティビティ アクティビティ` … 業務フローの順序
- 答え方: 「この業務は〜の BUC で構成され、各 BUC では〜という仕事（アクティビティ）が行われる」のように、**人と仕事の流れ**として説明する。

### B. システム／機能に関わる問合せ

- 観点: **UC** を中心にたどる。
- 使用する辺:
  - `#edge アクティビティ UC` … 仕事のうちシステム化される接点
  - `#edge UC 情報` … その UC がどの情報を扱うか
  - `#edge UC 画面` / `#edge UC イベント` / `#edge UC タイマー` … 外との接点
  - `#arrow 状態 UC` / `#arrow UC 状態` … UC が遷移させる状態
  - `#edge UC 条件` … 判断材料
- 答え方: 「このシステムでは UC〈…〉が〜の情報を更新／参照し、〜の画面・状態に影響する」と**システムの振る舞い**として説明する。

### C. システム全体の整合性／網羅性に関わる問合せ

- 観点: **情報 と UC の関わり**で網羅性を見る。
- チェック項目:
  - **孤立 UC**: `UC` が `#edge UC 情報` を持たない、または接続情報が極端に少ない。
  - **孤立情報**: `情報` が `#edge UC 情報` で参照されていない（操作されない情報）。
  - **片側欠落**: ある情報を**読む UC はあるが書き込む UC がない**（またはその逆）。
  - **状態の整合**: `状態モデル` の各状態に対して、遷移させる `UC`（`#arrow`）が揃っているか。
  - **コンテキスト境界**: `#child コンテキスト 情報/状態モデル/条件/バリエーション` が同一コンテキストに揃っているか。
- 答え方: 「情報 X は UC〈…〉が更新するが、参照する UC が見当たらない」のように、**事実 → 観察された不足／重複 → 想定される確認ポイント**の順で示す。

## このスキルの守備範囲（境界）

- **担当する**: 業務／システム／整合性・網羅性の問合せ回答（A/B/C 観点）、孤立 UC・孤立情報・片側欠落・状態整合の指摘。
- **担当しない**: 関連データの行ごとの逐語解説・変換表との照合中心の説明 → `rdra-relation-explain`。オブジェクトの変更・追加にともなう波及分析 → `rdra-impact-analysis`。

## 手順

1. **関連データを読む**: `1_RDRA/if/関連データ.txt` を Read。1 行目はシステム名として回答冒頭に使う。
2. **問合せを分類**: 上記 A / B / C のいずれかに振り分ける。曖昧なら確認する。
3. **対象オブジェクトを特定**: 問合せに含まれる名前を、関連データの `D 列`（`@@` の左右、`//` 区切りペア）から検索し、**どのモデルに現れるか**を確認する。
4. **辺をたどる**: 観点に対応する関係種別（`#child` / `#edge` / `#arrow` / `#attribute` / `#comment`）を絞り、必要な範囲で隣接ペアを列挙する。
5. **意味に翻訳**: たどった辺を `RDRA.md` の「モデル間のつながりの規則」に照らして自然文に直す。用語は `RDRA.md` の日本語ラベル（業務 / BUC / アクティビティ / UC / 情報 / 状態モデル / 状態 / コンテキスト / 条件 / バリエーション / アクター / 外部システム / 画面 / イベント / タイマー）を使う。
6. **類推は明示**: 関連データに直接書かれていないことを答える場合は「関連データには直接の記載なし。〜と類推される」と注記する。
7. **整合性の指摘（C 観点時）**: 観察された欠落・重複・断絶を、事実（行・ペア）に紐付けて示す。

## 出力テンプレート

```
# <システム名>: <問合せの要約>

## 種別
A. ビジネス / B. システム / C. 整合性・網羅性 のいずれか（複数可）

## 関連データから読み取れる事実
- 対象: <オブジェクト名>（モデル: <…>）
- 直接の関わり:
  - <#関係種別 モデル1 → モデル2>: <相手オブジェクト>（…）
  - …

## 説明（わかりやすい文章）
<業務観点なら「業務 → BUC → アクティビティ」、システム観点なら「UC とその接点」、
 網羅性観点なら「情報 ↔ UC の対応」を軸に、関連データの事実から平易に説明>

## 類推・補足（あれば）
- 関連データに直接の記載はないが、<モデル間の規則>から〜と考えられる。

## 確認したい点（C 観点や情報不足時のみ）
- <孤立 / 片側欠落 / 状態の不整合 など、関連データの該当行を引いて指摘>
```

## 出力上の注意

- **関連データに無いことは断定しない**。書かれているペアと、`RDRA.md` の規則から導けることだけを根拠にする。
- 用語は `RDRA.md` の日本語ラベルで統一し、「機能」「マスタ」のような曖昧語を使う場合は対応するモデル名（UC / 情報 など）を併記する。
- `@@` / `//` などの区切り記号は説明本文に残さない。
- 行・ペア数を一言添えると根拠が伝わりやすい（例: 「`#edge UC 情報` 行で 7 ペア該当」）。
- 関連データが大きい場合は、問合せに関係する関係種別だけに絞って読み解く。
